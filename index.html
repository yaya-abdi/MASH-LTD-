⚙️ System Setup

Configure Odoo before using any module — company setup, branches, users, email, and internal communication tools.

Companies, Branches & App Installation

2.1
Odoo home dashboard and app installation
›

The Odoo home dashboard shows all installed modules as app icons. You can install new apps from the App Store.

  1. From the home dashboard, click the Apps icon
  2. Browse or search for the module you need (e.g. Point of Sale, Manufacturing)
  3. Click Activate — the module installs automatically with all dependencies
  4. The new module icon appears on the home dashboard immediately
Note: Each module is independent with its own functionality. Installed modules are: CRM, Contacts, Calendar, Accounting, Projects, Inventory, Sales, and more.
2.2
Company setup and configuration
›
  1. Go to Settings → Users & Companies → Companies
  2. Open the company to edit
  3. Fill in: company name, address, phone, email, mobile, website, company ID (TRN), currency, logo
  4. For multi-company setups: create additional companies here; each can have its own currency and accounting
  5. The Document Layout option lets you customise how sales, delivery, and invoice printouts look
Note: For the MASH staging instance, one company has been created with MASH’s products, customers, suppliers, and chart of accounts already imported.
2.3
Branches
›

Odoo Enterprise supports branches within a company. A branch acts as a sub-entity under the parent company (MASH).

  1. Go to Settings → Users & Companies → Companies
  2. Open the parent company (MASH)
  3. Click the Branches tab to create or view branches
  4. Each branch has its own users and can operate independently while MASH remains the parent
  5. To switch between companies/branches: click the company name in the top bar and select the company to switch to
Note: When you switch to a branch, you see only that branch’s records. The parent company can see records of all branches.
2.4
Developer mode
›

Developer mode unlocks advanced technical menus including outgoing/incoming mail servers, scheduled actions, system parameters, and more.

  1. Go to Settings → General Settings
  2. Scroll down to find Developer Tools
  3. Click Activate the developer mode
  4. The page reloads — you now see a Technical menu in Settings and debug icons throughout the system
Warning: Developer mode exposes technical settings that can break the system if changed incorrectly. Only enable for admin users who know what they are doing.

Discuss & Calendar

3.1
Discuss — internal messaging
›

The Discuss module is Odoo’s internal communication tool — think of it as a built-in WhatsApp/Teams for your organisation.

  • Direct Messages — one-to-one chat with any internal user. Search by name and send messages with attachments, GIFs, emojis, or voice notes.
  • Channels (Groups) — create group chats like WhatsApp groups. Add multiple team members. Set access rights for each channel.
  • Inbox — shows all messages where you have been tagged or mentioned across the system
  • Starred messages — star important messages to find them easily later
  • History — full log of all messages, sales orders, CRM activities linked through the chatter
  • Video/voice calls — start a call or meeting directly from Discuss without leaving Odoo
Note: Discuss reduces email overhead for internal communication. All team conversations are tracked and searchable in one place.
3.2
Calendar — meetings and scheduling
›

The Calendar module manages all meetings and activities across the organisation. It can sync with Outlook and Gmail.

Create a meeting

  1. Go to Calendar → click on the date/time slot
  2. Enter meeting name, start time, end time, attendees
  3. Add URL (for Odoo internal meeting, Google Meet, or Teams link)
  4. Add description and save
  5. Drag and drop the event to reschedule it

Recurrent meetings

  1. When creating a meeting, enable the Recurrence option
  2. Set frequency: daily, weekly, monthly
  3. Select which days (e.g. Monday to Friday)
  4. Set end date — Odoo creates all recurring instances automatically

Outlook / Gmail sync

  1. Go to Settings → Calendar
  2. Enter the Client ID and Client Secret from your Google/Microsoft Azure app
  3. Once connected, meetings in Outlook or Google Calendar sync both ways with Odoo
Note: As a team leader you can view team members’ calendars to check availability before scheduling meetings.

Email Configuration

Email must be configured before Odoo can send or receive messages. Requires Developer Mode enabled first.

4.1
Outgoing mail server (sending emails)
›
  1. Enable Developer Mode (Settings → General Settings → Developer Tools)
  2. Go to Settings → Technical → Outgoing Mail Servers → New
  3. Enter a name (e.g. MASH Gmail SMTP)
  4. Set Authentication: Username/Password
  5. Enter SMTP server address (e.g. smtp.gmail.com)
  6. Enter username (email address) and password/app password
  7. Set connection encryption (SSL/TLS or STARTTLS) — port auto-fills
  8. Click Test Connection — should show “Connection successful”
  9. Save
Note: All system emails (quotations, invoices, reminders, notifications) will be sent from this configured email address.
4.2
Incoming mail server (receiving emails as records)
›

The incoming mail server allows Odoo to automatically create leads, tickets, or messages from emails received in a specific inbox.

  1. Go to Settings → Technical → Incoming Mail Servers → New
  2. Set server type: IMAP or POP3
  3. Enter server address, port, username, and password
  4. Set the action (e.g. create a CRM lead from incoming emails)
  5. Save and click Test & Confirm
Note: Once configured, when a customer replies to a quotation email or sends an inquiry, their reply appears directly in the chatter of the relevant record in Odoo — no need to manually copy emails.

›

A custom User Roles feature was added under Settings → Users & Companies → User Roles. Instead of manually setting access rights for each user one by one, you create a role with pre-configured rights and assign users to it.

  2. Name the role (e.g. Sales Manager, Warehouse User)
  3. Set all module access rights for this role
  4. Add the users who should have this role in the Users field
  5. Save — all listed users inherit the role’s access rights automatically
Custom: User Roles is not part of default Odoo. Built to simplify user management — no need to configure each user individually when roles change.

Contacts (Customers & Vendors)

In default Odoo there is no hard separation between a customer and a supplier — one contact can be both. MASH has custom fields to make the distinction explicit.

2.1
Customer vs vendor distinction
›
  • A custom boolean Is Vendor was added to the contact form. When enabled, the contact is treated as a vendor/supplier.
  • When Is Vendor is ON: the Other Info tab shows Vendor Group, Vendor Code, and Guarantor.
  • When Is Vendor is OFF: the Other Info tab shows Customer Group, Customer Code, and Guarantor.
  • A second field sets whether the vendor is local or international — this feeds into purchase module filtering.
Custom: Default Odoo has no Is Vendor boolean. Added for MASH to cleanly separate customers from suppliers and drive purchase type filtering.
2.2
Multiple addresses (invoice & delivery)
›

A contact can have different invoice and delivery addresses. These are managed as child contacts under the main contact.

  1. Open the contact → click Add in the Contacts & Addresses tab
  2. Set the type: Invoice Address or Delivery Address
  3. Fill in the address details and save
  4. When creating a sale order for this customer, Odoo shows the relevant invoice and delivery address fields
Custom: In default Odoo the address dropdown shows all addresses in the system. For MASH it was restricted so the dropdown shows only the addresses belonging to that specific customer.

3.3
Units of measure (sales UoM vs purchase UoM)
›

A product can have different units of measure for sales and purchasing. Enable Units of Measure in Sales Settings first.

  • Sales UoM: The unit shown to customers on quotations and invoices (e.g. square metres)
  • Purchase UoM: The unit used when ordering from suppliers (e.g. boxes/cartons)
  • Odoo converts between the two automatically using a conversion factor defined in the UoM settings
  • UoM categories must match — you can convert within a category (kg ↔ g ↔ tonne) but not across categories
MASH example: Tiles sold in square metres to customers, purchased in cartons/boxes from suppliers. Define both UoMs on the product and set the conversion factor. Odoo handles the maths. The pending question about different UoM on delivery vs invoice is being investigated — see Tab 10.
3.4
Somalian description & foreign name
›

Two custom text fields added to the product form:

  • Somalian Description: The product name or description in Somali language. Prints on sale orders, invoices, and delivery slips when the customer requires Somali text.
  • Foreign Name: Product name in any other language (e.g. Chinese for a Chinese supplier). Prints on purchase orders and delivery documents for that supplier.
Custom: Not in default Odoo. Added so that printouts can display the correct language name depending on customer/supplier requirements.
3.5
Routes, lead times & accounting
›
  • Routes: Define how the product moves through inventory (e.g. Buy, Manufacture, Drop Shipping). “Buy” means: take from stock → deliver to customer.
  • Customer Lead Time: Days it takes to deliver to the customer after the order. Updates the scheduled delivery date automatically on new orders.
  • Weight / Volume: Used to compute shipping costs when a delivery method with weight-based rules is applied.
  • Income Account: The revenue account for sales of this product. Can be set at product level, category level, or globally in accounting settings (product level takes priority).
  • Expense Account: The cost account for purchases or returns of this product.
  • Can Be Sold: Must be ticked for the product to appear in sale order product dropdowns.
  • Can Be Purchased: Must be ticked for the product to appear in purchase order product dropdowns.

Product Variants & Categories

4.1
Attributes and variants explained
›

A product variant is a specific version of a product defined by its attributes. Each combination of attribute values becomes its own separate tracked variant.

Example: Product = Shirt. Attribute 1 = Colour (Blue, Yellow). Attribute 2 = Size (Medium, Large). This creates 4 variants: Blue-Medium, Blue-Large, Yellow-Medium, Yellow-Large. Each variant has its own on-hand quantity, barcode, and can have its own price premium.

  1. Go to Sales → Configuration → Attributes and create attributes (e.g. Colour, Size)
  2. Add values to each attribute (e.g. Blue, Yellow; Medium, Large)
  3. Open the product form → Attributes & Variants tab
  4. Add the attributes — Odoo auto-generates all variant combinations
  5. Click Configure next to each attribute line to set price premiums per variant value if needed
  6. Click Product Variants button to manage each variant individually (quantity, barcode, etc.)
Note: Variants must be enabled in Sales → Configuration → Settings before the Attributes tab appears on products.
4.2
Buy in bulk, then split into colours
›

MASH’s scenario: buy a product as a single item (no attributes), then add colour variants afterwards and split the stock.

  1. Create the product without any attributes (e.g. “T-Shirt”)
  2. Purchase and receive 50 units — stock is recorded against the base product
  3. After receiving, add colour attributes to the product (e.g. Blue, Yellow)
  4. Odoo creates variant records: T-Shirt Blue, T-Shirt Yellow
  5. Use Inventory Adjustments to split the 50 units: set T-Shirt Blue to 30, T-Shirt Yellow to 20
  6. From this point onwards, each colour is tracked and purchased independently
Note: Once variants are added, future purchases will show the variants separately. The team confirmed this workflow is acceptable.
4.3
Importing and exporting variants
›

With many products and many colour/size variations, manual entry is impractical. Odoo supports bulk import and export.

  • From the product list or variant list, click the ⚙ Action button → Import Records
  • Download the Excel template, populate it with product names, attribute values, prices, and UoMs
  • Upload the completed file — Odoo creates or updates all products and variants
  • Similarly use Export to download all current products/variants into Excel for review or bulk editing
Note: Import/export is available on all list views across Odoo, not just products.
4.4
Product categories
›

Found in Inventory → Configuration → Product Categories (or Sales → Configuration).

  • Name & Parent Category: Create a hierarchy (e.g. All → Building Materials → Tiles)
  • Costing Method: Standard Price, Average Cost (AVCO), or First In First Out (FIFO). Applied to all products in this category.
  • Inventory Valuation: Manual or Automated. Automated posts journal entries every time stock moves.
  • Income Account: Default revenue account for all products in this category (can be overridden at product level)
  • Expense Account: Default cost account for this category
Important: Landed costs only work with AVCO or FIFO costing. If product categories are still set to Standard Price, update them before go-live.

Pricelists

Pricelists allow different selling prices per customer or customer group. Enable Pricelists in Sales → Configuration → Settings first.

5.1
Simple pricelist rules
›
  1. Go to Sales → Configuration → Pricelists → New
  2. Enter a name (e.g. Wholesale Customers, Good Customers)
  3. Set currency
  4. Under Rules, click Add a line
  5. Select the product or product category this rule applies to
  6. Set Min. Quantity: the rule only activates when the customer orders at least this many units
  7. Set the Fixed Price to apply
  8. Optionally set a Start Date and End Date for time-limited promotions
  9. Save
  10. Assign this pricelist to the customer in their Sales & Purchase tab
Example: Product X normally costs 150. For customers on the “Wholesale” pricelist, if they buy at least 2 units, the price drops to 100. When 2 or more units are added to a sale order, the price auto-updates to 100.
5.2
Advanced pricelist rules
›

Enable Advanced Price Rules in Settings to unlock more options.

Computation MethodHow it works
Fixed PriceSet a specific absolute price. No formula.
DiscountApply a % discount off the product’s sales price (e.g. 10% off for this pricelist)
FormulaCompute price from cost or another pricelist, then add a surcharge or margin. Flexible for complex pricing strategies.

Rules can be applied to: All Products, a specific Product Category, a specific Product, or a specific Product Variant.

User-based discount limit (custom)
›

  • A new Discount Limit field appears on each user’s form (in Settings → Users)
  • Set the maximum discount % per user (e.g. administrator can give up to 10%)
  • If a user tries to apply a global discount above their limit, the system shows a warning and prevents it
Custom: Not in default Odoo. Built to prevent sales staff from giving excessive discounts without approval. Note: trainer observed a potential module update issue in the session — to be verified.

Sales Commission

Sales commission is a fully custom feature. Default Odoo has no commission system.

7.1
Commission plan types
›
Plan TypeHow it works
StandardCommission is calculated on the total sale order value. A threshold limit applies — only sales above the limit earn commission. One plan per salesperson.
Product/Category BasedCommission is applied per specific product or product category at different rates. Useful when different product lines earn different commission percentages.
7.2
Setting up a commission plan
›
  1. Go to Sales → Configuration → Sales Commission
  2. Click New
  3. Select the Salesperson (each salesperson can only have one plan)
  4. Choose plan type: Standard or Product/Category Based
  5. For Standard: set the Commission Limit (threshold) and the Commission % (e.g. limit = 1,000; earn 10% on any sale above 1,000)
  6. For Category-based: add lines specifying product / product category / subcategory and the corresponding % rate
  7. Save
7.3
When is commission computed?
›

Configure the trigger in Sales → Configuration → Settings → Sales Commission Rule:

TriggerWhen commission is calculated
Sale Order ConfirmedImmediately when the SO is confirmed, before delivery or invoice
Invoice PostedWhen the invoice is confirmed/posted in accounting
Payment ReceivedOnly after the customer pays the invoice
7.4
Commission lines and payout
›

Once commission is computed, it appears as a Commission Line under Sales → Commission Lines.

  • Filter by salesperson and month to see what is owed
  • Lines in “To Invoice” state need to be paid out
  • Option 1 — Vendor Bill: Select the lines → click Create Invoice → generates a vendor bill to pay the salesperson their commission
  • Option 2 — Payroll: Commission lines automatically carry into the monthly payslip calculation when payslips are run for the employee. The accumulated commission for that month is included in the payslip.
Custom: The entire commission system is custom-built. The payroll integration (commission appearing in payslips) is also a custom link between the Sales and HR/Payroll modules.

8.1
Creating a quotation
›
  1. Go to Sales → Orders → Quotations → New (or create from a won CRM opportunity)
  2. Select the customer — invoicing address and delivery address auto-populate from the contact
  3. Set expiration date and payment terms (auto-fills from customer if pre-set)

  8. Save
Note: Pricelists allow different prices per customer. Define a pricelist (e.g. Good Customer — 100 instead of standard 150) and assign it to the customer contact. It auto-fills on new quotations for that customer.
8.2
Sending the quotation and confirming
›
  1. Click Send by Email — a wizard opens with a pre-filled email template including the quotation PDF
  2. Verify recipient email address, subject, and body message
  3. Click Send — quotation status moves to “Quotation Sent”
  4. When customer confirms, register the advance payment (if mandatory setting is enabled)
  5. Click Confirm — quotation becomes a Sales Order and is locked for editing
  6. A delivery order is automatically created
Note: A product price limit warning appears if the price entered differs from the product’s configured minimum/maximum price. You can proceed or stop based on the warning.
8.3
Delivery
›
  1. From the confirmed Sales Order, click the Delivery smart button (top right)
  2. The delivery shows all products, demanded quantities, and available stock
  3. Add carrier details and tracking reference if applicable
  4. Print delivery slip or product labels if needed
  5. Get customer signature using the Sign button — a signed delivery slip PDF is auto-attached to the chatter
  6. Click Validate — stock quantity reduces and delivery moves to Done
Note: The delivery slip shows who delivered, what was delivered, quantities, and the customer’s signature — useful as proof of delivery.
8.4
Returns
›
  1. Open the validated delivery → click Return
  2. Select which products and how many to return
  3. Set the return location (stock or a separate returns/damaged location)
  4. Click Return — a new reverse delivery is created
  5. Validate the return delivery — stock is added back
  6. The Sales Order now shows 2 deliveries: original + return
8.5
Invoicing and payment
›
  1. From the Sales Order, click Create Invoice
  2. Invoice details auto-populate from the SO — products, quantities, prices
  3. Click Confirm — invoice is posted and a journal entry is created
  4. Go to Accounting → Customers → Payments → New to register payment
  5. Select customer, amount, journal (bank account), TRN number
  6. Confirm payment — the invoice outstanding credit is matched and moves to Paid
Note: Split payments are supported — a customer can pay part from one bank and part from another. Create two separate payments for the same invoice and match both.
8.6
Sales reporting
›

2.1
Receipts (incoming products from vendors)
›

A receipt records incoming products from a vendor into your warehouse. It is created automatically from a confirmed purchase order, or can be created manually.

  3. Set destination location (e.g. Shelf 1, WH1/Stock)
  4. Set scheduled date and add source document reference (your copy of the PO number)
  5. Add products and demanded quantities
  6. Click Save, then Validate — products are added to your stock

2.2
Deliveries (outgoing products to customers)
›

Deliveries record outgoing products shipped to customers. Created automatically from confirmed sales orders, or manually.

  1. Go to Inventory → Operations → Deliveries → New
  2. Set the destination location (customer location)
  3. Set delivery method: transporter or direct delivery
  4. Select vehicle/fleet if applicable
  5. Set source location (which warehouse stock to pick from)
  6. Add products and demanded quantities
  7. Click Check Availability — red shows insufficient stock
  8. Get customer signature if required, then Validate
Note: When validated, stock on hand decreases by the delivered quantity. A signed delivery slip is automatically attached to the chatter.

Delivery features

  • Tracking reference — add carrier tracking number for shipment monitoring
  • Customer signature — capture a digital signature before validating; generates a signed delivery slip
  • Print delivery slip — printable document for the customer
  • Print product labels — barcode labels for each unit being shipped
  • Pack into package — add products to a named package before shipping
2.3
Internal transfers (moving stock between locations)
›

Internal transfers move products between locations within the same company — e.g. from WH1/Stock to WH2/Stock, or from one shelf to another.

  1. Go to Inventory → Operations → Transfers → New
  2. Set operation type to Internal Transfer
  3. Set source location (where to take stock from)
  4. Set destination location (where to move it to)
  5. Add products and quantities
  6. Click Mark as Todo, then Validate
Note: After validation, the on-hand quantity at the source location decreases and increases at the destination. Total company inventory remains the same.
Warning: Internal transfers only work within the same company. Transfers between separate branch companies require inter-company transactions. See Tab 8 for the inter-branch scenario.
2.4
Inventory adjustments (physical inventory / stock count)
›

Use inventory adjustments when the actual physical count in your warehouse differs from the system quantity. This corrects the system to match reality.

  1. Go to Inventory → Operations → Physical Inventory
  2. Create a new adjustment
  3. Add the product and the location to count
  4. Enter the counted quantity (what you physically see in the warehouse)
  5. The system shows the difference between system quantity and counted quantity
  6. Click Apply All — the system quantity is updated to match your count
Note: After applying, the product page shows the adjustment as a separate stock entry so the history is clear — you can always trace why the quantity changed.
2.5
Scrap orders (damaged or unusable products)
›

Scrap orders move damaged or unusable products out of usable stock into a dedicated scrap location.

  1. Go to Inventory → Operations → Scrap
  2. Click New
  3. Select product and quantity to scrap
  4. Select source location (e.g. WH2/Stock)
  5. Add source document reference if available
  6. Optionally mark as a replenishment need
  7. Click Validate — product moves to the virtual scrap location
Note: The scrapped quantity is visible on the product page under stock movements. Odoo creates a virtual scrap location automatically — no setup needed.
2.6
Landed costs
›

Landed costs add shipping, customs, and other charges to the cost of received products. Go to Inventory → Operations → Landed Costs.

  1. Reference the receipt (transfer) you want to apply costs to
  2. Add carrier/vendor details
  3. Add cost lines (transport, customs, insurance, etc.)
  4. Choose split method: equally, by quantity, by weight, or by value
  5. Click Validate — costs are distributed across product units
Warning: Landed costs require Average Cost (AVCO) or FIFO costing method on the product category. Standard Price does not support landed costs.

Inter-Branch Transfers

This was a key discussion point in the session. Below covers how Odoo handles branch transfers and the agreed approach for MASH.

How branches work in Odoo

When you create a branch in Odoo, it is treated as a separate company entity with the parent company (MASH) as its owner. This means:

  • Each branch can have its own users, contacts, transactions, reports
  • A branch appears as a contact/partner in the parent company’s system
  • Odoo does NOT treat stock movement between separate companies as an internal transfer — it treats it as a sale/purchase between companies

Default Odoo option: inter-company transactions

8.1
How inter-company transactions work
›
  1. Go to Settings → General Settings → Inter-Company Transactions → Enable
  2. Choose what to synchronize: invoices/bills, sales orders, or purchase orders
  3. At the branch level also enable the synchronization option
  4. Create a sales order in Company A (MASH) for Branch 1 as the customer
  5. Odoo automatically creates a purchase order in Branch 1 corresponding to that sale
  6. The vendor reference on the Branch 1 PO shows the MASH SO number
Limitation: This approach creates accounting entries (sale and purchase) between branches. MASH’s requirement was for a zero-accounting-impact internal transfer — just moving stock between branches with no sales/purchase effect.

MASH’s agreed approach: Material Request via warehouse

8.2
Warehouse-based inter-branch transfers (agreed approach)
›

Instead of creating branches as separate companies, treat each branch as a separate warehouse under the same company. Then use the Purchase Request → Material Request workflow to transfer between warehouses.

8.3
Pending custom requirement: self-approval prevention
›

MASH raised a concern: a branch manager who is both the requester and the approver could approve their own material request. This needs to be blocked.

MASH’s requirement: A user with Purchase Request Manager access should be able to approve requests submitted by others — but NOT approve their own requests.

Agreed technical solution: Add a validation check in code: if the user trying to approve is the same as the user who created the request, raise a validation error: “You cannot approve a material request you created.”

Status: This customization is not yet built. It needs to be developed before go-live to ensure the approval workflow has proper segregation of duties.

Vendor Evaluation

4.1 How it works

Vendors are evaluated per quarter on four parameters:

  • Delivery on time — did goods arrive by the promised date?
  • Correct items — were the right products delivered?
  • Correct quantities — were the ordered quantities fulfilled?
  • Meets due date — overall commitment reliability

Scores are entered per parameter. The system calculates a final evaluated point for each quarter. All four quarters are aggregated into a yearly score.

4.2 Running an evaluation

  1. Go to the vendor contact form
  2. Select the Year and Quarter (Q1, Q2, Q3, Q4)
  3. Enable the evaluation and enter scores for each parameter
  4. Click Calculate — system generates Final Points and Final Evaluated Points
  5. Repeat for all four quarters to build the full yearly score

4.3 Yearly scoring and PO blocking

The system aggregates all four quarterly evaluations into a yearly score. If the yearly evaluated score is below 2.5:

  • The vendor can still receive RFQs
  • The RFQ can be sent to the vendor by email
  • Confirming the PO triggers a validation error and blocks the order
Custom: PO blocking on vendor evaluation score is a custom validation. Threshold is 2.5. Update the vendor's quarterly scores to re-enable PO creation if the vendor improves.

Tax computation methods

MethodHow it works
Group of TaxesCombines multiple taxes into one (e.g. VAT + municipal tax)
Fixed AmountA flat amount added per unit regardless of price
Percentage of Price% of the tax-excluded price (most common)

Tax included vs tax excluded

4.2
Bill of Materials (BOM)
›

The Bill of Materials defines what components and operations are needed to produce a finished product. It is the foundation of every manufacturing order.

  1. Go to Manufacturing → Products → Bills of Materials → New (or create from the product form)
  2. Select the finished product to be produced

  4. Add operations — define each production step (e.g. Plan & Design, Gather Materials, Cut & Assemble, Finish)
  5. Assign each operation to a Work Center and set the planned duration

Work Centers
›

Work centers are physical or logical stations where manufacturing operations are performed (e.g. cutting station, assembly line, painting booth).

  1. Go to Manufacturing → Configuration → Work Centers → New
  2. Enter work center name
  3. Set cost per hour (per work center, or per employee)
  4. Set electricity cost per hour
  5. Assign to relevant operations in the BOM
Note: Work center costs (labor + electricity) are either taken from the work center directly or entered manually on each manufacturing order — configurable in Settings.
4.4
Manufacturing Order
›
  1. Go to Manufacturing → Operations → Manufacturing Orders → New
  2. Select the product — components auto-populate from the BOM
  3. Set scheduled start date — end date is calculated automatically based on operation durations
  4. Review the Work Orders tab to see all operations and their work centers
  5. Set the finished product storage location if different from default
  6. Add actual labor cost and actual quantity used once production begins
  7. Click Confirm, then Plan, then Produce
Note: The manufacturing order overview shows: unit cost, total components used, total operations, and a full cost breakdown (material, labor, electricity).
4.5
Work Orders
›

Work orders are the individual steps within a manufacturing order, each assigned to a specific work center and person.

  1. Open a confirmed manufacturing order → go to the Work Orders tab
  2. Each operation from the BOM appears as a separate work order
  3. Assign the work order to a specific employee
  4. Click Start to begin the timer for that work order
  5. Pause and Resume as needed during the work
  6. If an issue occurs, click Block and enter a reason (e.g. equipment failure)
  7. When complete, click Mark as Done
Note: Work orders track actual time spent vs planned time. The actual duration is recorded automatically via the timer and feeds into cost analysis.
4.6
Shop Floor
›

Shop Floor is a simplified view for production workers who do not have access to the full Odoo backend. Workers can see their assigned work orders, complete quality checks, and track progress without needing backend access.

  1. Enable Shop Floor in Manufacturing → Configuration → Settings
  2. Workers open the Shop Floor view on a tablet or workstation
  3. They see their assigned work orders and click to start work
  4. Quality checks, alerts, and improvement suggestions can be added from here
  5. Once a step is marked Done, the next work order in sequence becomes active
  6. Workers can also move defective items to scrap from the Shop Floor
Note: Shop Floor is ideal for factory workers who only need to see their specific tasks — no access to orders, inventory, or accounting required.
4.7

Scrap orders

  1. In the manufacturing order, click Scrap

  3. Confirm — the item moves to the scrap location and is removed from usable stock

Unbuild orders

If a finished product needs to be disassembled back into its components (e.g. a scrapped product whose parts are still usable):

  1. In the completed manufacturing order, click Unbuild
  2. Select quantity to unbuild
  3. Confirm — all components that were used are returned to stock
  4. Unbuild orders can also be created independently without a manufacturing order link
Note: Unbuild can be used both on finished products and on products in the scrap location — useful for recovering valuable components from defective goods.

Multi-Branch HR Setup

MASH raised a question: "Even though we have one company with different branches, does it come under one HR manager / one HR department?"
